1099 Contractor · First UX hire

This role expanded the scope of my responsibilities as a designer and gave me the opportunity to build a design framework from scratch.

NextHealth Technologies is a healthcare analytics platform serving health plans, ACOs, and employer groups. When I joined as the first UX hire, the product had grown organically through engineering-led development — functional, but without a cohesive design language or systematic approach to interaction. My work focused on establishing that foundation while delivering immediate improvements to key product areas.

Initial Assessment

The platform had accumulated years of interface debt. Components were inconsistent across modules, navigation patterns conflicted, and there was no shared vocabulary between design and engineering. Before any redesign work could begin, I needed to map what existed — conducting a full audit of the product's UI patterns, documenting inconsistencies, and identifying which areas had the highest user-facing impact.

Stakeholder interviews surfaced a recurring theme: users found the platform powerful but difficult to navigate, particularly when moving between programs and reports. The audit confirmed this — there were four distinct navigation paradigms operating simultaneously with no clear hierarchy.

UI audit findings — component inconsistency mapping

Addressing the Platform UI

With Material UI already embedded in the engineering stack, I made the decision to build on top of it rather than replace it. The goal was to extend the Material component library with a cohesive visual layer specific to NextHealth — establishing consistent color use, typographic hierarchy, density standards, and interaction states that could be documented and handed off systematically.

The before/after below shows the shift from an ad-hoc component approach to a structured design language. This work reduced design-to-dev handoff friction significantly and gave the engineering team a clear reference for new feature work.

Before — legacy component patterns After — unified component library

Navigating Platform Navigation

The navigation redesign was among the most complex challenges of the engagement. The platform served different user roles — care managers, analysts, and administrators — each with distinct workflows and access requirements. The existing navigation made no distinction between these paths, surfacing everything to everyone.

I worked closely with the product team to define a role-aware navigation model that surfaced relevant features contextually. The restructure reduced top-level navigation items by roughly 40% while improving discoverability of key features across user testing sessions.

Navigation architecture — before and after restructure

Reusable Patterns

A significant output of this engagement was a documented pattern library built in Figma — covering form elements, data table variants, filter patterns, status indicators, and modal workflows. These patterns were tied directly to Material UI component names and variants, so engineers could cross-reference the design file without ambiguity.

Beyond components, I documented interaction patterns: how empty states behaved, when to use inline vs. modal editing, how error states should surface across different contexts. This gave the team a shared design language that extended past my direct involvement.

Figma component library — pattern documentation

Closing Thoughts

This engagement taught me a lot about the shape of design influence in organizations that are new to it. With no prior design culture, nearly every conversation was also an education — making the case for process, for documentation, for slowing down to go faster. The work that had the most lasting impact wasn't any single screen, but the systems and shared language left behind when the engagement concluded.
